## P2：データ完全性

### P2-1. exportData/importData が一部キーを含まない（med_list 等）
- 症状：エクスポートされるJSONに med_list 等が含まれず、バックアップが不完全。
- 推定原因：export/import 対象キーのリストに新しいキーが追加されていない。
- 依存関係：★JSONバックアップ整理計画の「前」に直すこと。
  不完全なバックアップのまま整理すると、薬リスト等が欠けた状態で古い分を削除し、
  復元時に取りこぼす事故が起こり得る。整理着手より先に対応する。
- 関連：data-model.md（保存場所と形式・キー名の使い分け）。

## R：要調査・要観察（優先度未確定）

### R-1. next_visit / consultation_summary のレガシー形式（文字列・idなし）マイグレーション
- 背景：両キーのマージキー関数（`nvKey`/`csKey`）は要素が文字列または `id` なしオブジェクトの
  場合に `'text:' + テキスト` という合成キーにフォールバックする。一方 `filterByTombstone` は
  `item.id`（無ければ `item.date`）しか見ないため、idを持たないレガシー形式の要素は
  tombstoneでの削除照合ができない可能性がある（2026-06-19調査で判明）。
- 着手前に：実際にidなしのレガシー形式データが現存するか（localStorage / GASの実データ）を
  先に調査してから対応方針を決める。存在しなければ対応不要。
- 関連：sync.md（キーごとのマージ戦略）。

### R-2. push失敗の静かな取りこぼし（要観察）
- `pushToCloud` は失敗時 `catch {}` で握りつぶし、呼び出し元・画面のどちらにも何も返さない。
- 今のところ実害（取りこぼし）が頻発している様子はないが、「チェックしたのに別端末に来ない」
  系の報告があったら、まずGAS側 `chk_<日付>` の値を確認 → 無ければflush取りこぼしを疑う、
  の順で調査する（詳細は troubleshooting.md）。
- 恒久対応（リトライ導入等）が必要か、しばらく様子を見てから判断する。

- 調査の結果、tombstone自体は8カテゴリで実装済みと判明（「薬管理側に実装済み前例」という
