## A. セキュリティ

### S1（深刻度：高）GAS `doGet`/`doPost` に認証機構が一切ない

- 該当：`index.html` 1337-1345行（`pushToCloud`）、1358-1369行（`flushChkPending`）、
  1433行（`syncFromCloud`）、1736-1740行（`pushAllToCloud`）
- 削除前の `gas_code.js`（コミット b7bdb71、55行）を確認したところ、`doGet`/`doPost` は
  トークン・パスワード等の検証を一切行わず、リクエストを受けたら即座に読み書きしている。
- クライアント側（index.html）のfetch呼び出しも、`Authorization` ヘッダーや秘密パラメータを
  一切付与していない。つまり **GAS URLさえ知っていれば、第三者が全データを読み取り／
  任意の値に書き換え可能**（どのキーにも `doPost` で `{key, value}` を送るだけでよい）。
- 現在のGAS側コードは未確認（リポジトリにファイルが存在しないため）だが、クライアント側の
  呼び出し方が変わっていない以上、認証が追加されていない限り同じ状態と推測される。
- 推奨対応方針：
  1. GAS側で共有シークレット（例：`doPost`/`doGet` の `e.parameter.token` 等）を必須化し、
     一致しない場合は403相当のレスポンスを返す。
  2. クライアント側は `gas_url` とは別に `gas_token` をlocalStorageに保存し、fetch時に付与する。
  3. トークンはURLクエリではなくPOSTボディに含める（アクセスログ等へのURL露出を避ける）。
  4. 対応後は現行デプロイのURLをローテーション（再デプロイ）し、旧URLを失効させる。

### B5（深刻度：低）自動バックアップ・体重グラフの日付境界計算がUTCベース

- 該当：`autoBackup()` 1810行、`_filterWeightData()` 5264行

```js
const cutoffStr = cutoff.toISOString().slice(0, 10); // UTC日付
```

- アプリの他の日付処理（`todayKey()` 1254-1257行 等）は一貫して
  `getFullYear()/getMonth()/getDate()`（ローカル時刻＝JST）を使っているのに対し、
  この2箇所だけ `toISOString()`（UTC）を使っている。JSTはUTC+9のため、
  ローカル日付の午前0時〜8時59分の間は、UTC日付が「前日」のまま計算される。
- 影響：
  - `autoBackup()`：7日より古いバックアップの削除判定が、深夜0時〜9時の間だけ
    実際の意図より最大1日分ずれる（消すべきものを消さない／早めに消す）。
    バックアップは複数日分残るため実害は小さい。
  - `_filterWeightData()`：体重グラフの「直近30日／90日」表示で、境界の日の体重記録が
    深夜0時〜9時台の閲覧時に限り、期待と異なる範囲で表示される可能性がある
    （表示のみでデータ欠損はない）。
- 推奨対応：`todayKey()`と同様にローカル日時ベースの計算に統一する。
- 【対応済み 2026-07-04／コミット f88dfec】2箇所とも `todayKey()` と同じローカル日時の
  組み立てに置き換え（各1行のみの修正。cutoff計算・削除ループ・フィルタ条件は不変）。

## 修正の推奨順序

深刻度と修正の容易さ（影響範囲の狭さ）を基準に、以下の順序を提案する。

1. **S2（日誌のXSS）** — 最優先。実運用可能なXSSで、修正は`renderDiary`内の6箇所に
   `escHtml()`を追加するだけ。既存パターンの単純な踏襲で低リスク。
2. **S2b（`escHtml`の属性コンテキスト問題）** — S2と同時に対応するのが効率的。
   該当1箇所を`_esc()`に差し替えるだけの最小修正で完了する。
3. **S1（GAS認証なし）** — 深刻度は最も高いが、GAS側コード（Apps Scriptエディタで直接編集・
   このリポジトリには存在しない）とクライアント側の両方に変更が必要で、設計判断
   （トークンの受け渡し方法等）を先に確定させる必要がある。CLAUDE.mdの運用ルールに従い
   「実装前にモックアップ・方針を提示して承認を得る」対象。独立したセッションを推奨。
4. **B1（tombstone早期returnの5項目への拡大）** — 中程度の影響。`health_notes`の
   既存パターンを他5項目に横展開する対応で、設計判断はほぼ不要（前例踏襲）。
   ※着手前に方針をerisaと相談してから実装すること。論点＝「isEmptyガード（GAS誤空からの
   保護）の目的」と「tombstone適用による削除伝播」の両立方法。syncFromCloud の心臓部
   6ブロックを触るため、複数端末での削除伝播シナリオでの動作確認が必要。

   try-catchを追加するのみ。
6. **B5（UTC日付境界のずれ）** — 影響は小さいが修正は簡単（2箇所を`todayKey()`と
